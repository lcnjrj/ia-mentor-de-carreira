# Roadmap: Technical Writer (Escritora Técnica)

Olá! Recebi suas informações do entrevistador. Vejo que você escolheu Technical Writer (Escritora Técnica) e tem 20 horas por semana para estudar.  Perfeito! Vamos unir sua vasta experiência em sistemas Linux com sua habilidade nata de organização para criar documentações de alta precisão técnica. Vou montar agora seu plano completo personalizado... 

---

## Visão do Dia a Dia

Como é o trabalho de uma Technical Writer:

* 
**Docs-as-Code**: Escrever documentação técnica usando as mesmas ferramentas que desenvolvedores (Git, Markdown, VS Code). 


* 
**Entrevistas com Especialistas (SMEs)**: Conversar com engenheiros para entender como uma nova funcionalidade funciona "debaixo do capô". 


* 
**Testes de Código**: Rodar scripts e APIs pessoalmente para garantir que o passo a passo da documentação realmente funciona. 


* 
**Curadoria e Taxonomia**: Organizar a arquitetura da informação para que o usuário encontre o que precisa em poucos cliques. 


* 
**Documentação de APIs**: Estruturar guias de referência para desenvolvedores, detalhando parâmetros, retornos e exemplos de uso.

## Mapa de Skills

### Core Skills (essenciais)

* 
**Escrita Técnica**: Domínio de clareza, concisão e tom de voz técnico (Português e Inglês). 


* 
**Linguagens de Marcação**: Markdown e reStructuredText. 


* 
**Git/GitHub**: Gerenciamento de versões de documentos e fluxos de Pull Request. 


* 
**Domínio de Sistemas**: Sua base sólida em Linux e Shell Script para documentar infraestrutura. 



### Nice-to-Have (complementares)

* 
**IA para Documentação**: Uso de LLMs para revisão gramatical e geração de rascunhos de guias rápidos. 


* 
**UX Writing**: Princípios de microcopy para tornar interfaces mais intuitivas. 



### Ferramentas e Tecnologias

* 
**Static Site Generators (SSG)**: Docusaurus, Hugo ou MkDocs. 


* 
**APIs**: Swagger/OpenAPI (conhecer a estrutura para documentar). 


* 
**Linha de Comando**: Automação de builds de documentação via scripts.

## Roadmap de 90 Dias

**Adaptado para: 20h/semana**

### Mês 1 - Fundamentos e Docs-as-Code

* 
**Semana 1-2**: Princípios da Escrita Técnica: Estudo de guias de estilo (Google Developer Documentation Style Guide ou Microsoft Writing Style Guide). Markdown Profissional: Dominar a sintaxe e organização de arquivos .md. 


* 
**Semana 3-4**: Git para Escritores: Fluxo de trabalho de documentação integrada ao código. Estruturação de Informação: Aplicar suas técnicas de pesquisa e clipping para criar taxonomias de documentação. 



### Mês 2 - Ferramentas e APIs

* 
**Semana 5-6**: Static Site Generators: Criar um site de documentação funcional usando MkDocs ou Docusaurus. Documentação de Shell Scripts: Criar manuais (man pages) e guias de usuário para ferramentas de terminal. 


* 
**Semana 7-8**: Documentação de APIs: Aprender a ler especificações OpenAPI (Swagger). Integração de IA: Explorar como documentar sistemas que utilizam Inteligência Artificial. 



### Mês 3 - Portfólio e Mercado

* 
**Semana 9-10**: Projeto de Portfólio: Focar na qualidade da explicação técnica. LinkedIn e Networking: Conectar-se com comunidades de Technical Writing (como a Write the Docs). 


* 
**Semana 11-12**: Soft Skills para Entrevistas: Focar na maturidade e na capacidade de aprendizado contínuo. Preparação de Cases: Como explicar sua transição de carreira focando em "entender para explicar".

## Projeto de Portfólio

**Título**: The Linux Automation Manual 

**O que fazer**: Documentar profissionalmente um sistema de automação em Shell Script (pode ser o seu rastreador de tempo ou um utilitário de sistema). Você criará um guia completo contendo: Guia de Início Rápido, Referência de Comandos, Troubleshooting e Contribuição. 

**Entregáveis**:

* Repositório no GitHub com a documentação em Markdown. 


* Site estático (GitHub Pages) hospedando a documentação. 


* Guia de API (se o script tiver interações externas). 



**Critérios de Aceitação**:

* Zero erros gramaticais. 


* Instruções que funcionam "de primeira" ao serem testadas em um terminal Lubuntu limpo. 


* Uso de elementos visuais (diagramas representados em código ou imagens). 



> 
> **Dica**: Use sua facilidade em organizar informações para criar uma seção de "Glossário Técnico" que ajude iniciantes a entenderem termos complexos de Linux.

## Roteiro de Entrevistas

1. **Como lidar com a resistência de desenvolvedores?** Diga que você atua como uma facilitadora. Você "traduz" o que eles fazem para o papel, reduzindo o número de perguntas repetitivas que eles recebem. 


2. **Como sua experiência com clipping ajuda?** Explique que a escrita técnica é 70% pesquisa e 30% escrita. Sua habilidade em sintetizar grandes volumes de dados garante que a documentação seja precisa. 


3. 
**Como documentar o que não domina?** Fale sobre sua base em Linux/Shell Script que permite ler o código para entender a lógica básica, seguida de testes práticos e entrevistas estruturadas. 


4. 
**Technical Writer ou Desenvolvedora?** Destaque que sua paixão está na intersecção entre a lógica do código e a clareza da comunicação para outros seres humanos. 


5. **Trabalho remoto e assíncrono?** Mencione sua organização pessoal e familiaridade com ferramentas de colaboração. A documentação exige foco e autonomia.
